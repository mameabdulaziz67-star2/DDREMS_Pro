═══════════════════════════════════════════════════════════════
🗄️ DATABASE CONNECTION
═══════════════════════════════════════════════════════════════

Table Used: property_documents

Columns:
  • id - Document ID
  • property_id - Links to properties table
  • document_name - Name of document
  • document_url - Base64 encoded document (LONGTEXT)
  • document_type - Type (title_deed, survey_plan, etc.)
  • access_key - 8-character access key
  • is_locked - Lock status (0 or 1)
  • uploaded_by - User who uploaded
  • created_at - Upload timestamp

API Endpoints Used:
  • GET /api/property-documents/property/:propertyId
  • POST /api/property-documents/verify-access
  • PUT /api/properties/:id/verify

═══════════════════════════════════════════════════════════════
🔑 ACCESS KEYS
═══════════════════════════════════════════════════════════════

How Access Keys Work:

1. Owner uploads document
   → System generates 8-character key (e.g., "A1B2C3D4")

2. Customer requests access
   → Request sent to owner

3. Owner approves request
   → Sends key to customer

4. Customer enters key
   → Document opens if key is correct

5. Property Admin can view
   → Needs access key like customers
   → Or owner can provide key directly

═══════════════════════════════════════════════════════════════
⚠️ IMPORTANT NOTES
═══════════════════════════════════════════════════════════════

1. Property Admin needs access keys to view documents
   (Same as customers - for security)

2. Verification buttons work immediately
   (No need to view documents first)

3. Status changes are permanent
   (Use carefully - confirm before clicking)

4. Documents are stored as base64 in database
   (LONGTEXT column - up to 10MB per document)

5. Locked documents cannot be viewed
   (Owner must unlock first)
